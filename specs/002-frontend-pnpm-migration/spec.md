# Feature Specification: Frontend pnpm Migration

**Feature Branch**: `002-frontend-pnpm-migration`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "As an engineer I want to use technologies that are on the vanguard of development. Because of that I want to use pnpm as my package manager for my node and react based projects. I want to add the specifications of the project, so we must update the AGENTS.md for future agentic development context. Therefore I want the frontend to be migrated to use pnpm. We must update the Docker images generation and make sure that everything is working before we close the migration. As a last point lets remove the previous files dependencies from the previous package management."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Developer installs frontend dependencies with pnpm (Priority: P1)

A developer clones the repository and runs `pnpm install` inside the `frontend/` directory. All dependencies resolve correctly from the `pnpm-lock.yaml` lockfile and the `node_modules` tree is populated without errors.

**Why this priority**: This is the foundational step. If local installation does not work, no other development or build activity is possible.

**Independent Test**: Can be fully tested by deleting `node_modules`, running `pnpm install`, and verifying zero errors and a complete `node_modules` folder with a `pnpm-lock.yaml` present.

**Acceptance Scenarios**:

1. **Given** a clean checkout with no `node_modules`, **When** `pnpm install` is run inside `frontend/`, **Then** all dependencies install successfully with no errors and a `pnpm-lock.yaml` is present.
2. **Given** the migrated repository, **When** the repository is inspected, **Then** `package-lock.json` no longer exists in the `frontend/` directory.

---

### User Story 2 - Developer runs local development server with pnpm (Priority: P2)

A developer starts the local development server using the pnpm-managed scripts and the frontend application starts correctly, hot-reloading on file changes.

**Why this priority**: The developer workflow must remain uninterrupted after the package manager change.

**Independent Test**: Run `pnpm run dev` inside `frontend/` and verify the application is accessible at `http://localhost:5173`.

**Acceptance Scenarios**:

1. **Given** dependencies installed via pnpm, **When** `pnpm run dev` is executed, **Then** the Vite development server starts and the app is reachable at `http://localhost:5173`.
2. **Given** the running development server, **When** a source file is edited, **Then** hot-module replacement reflects the change without a full page reload.

---

### User Story 3 - Docker image builds and runs successfully using pnpm (Priority: P3)

The Docker image for the frontend is built using pnpm instead of npm. The resulting container starts and serves the application correctly.

**Why this priority**: The containerised workflow (used in Podman Compose) must work end-to-end after the migration.

**Independent Test**: Build the frontend Docker image with `podman build` and start a container, then verify the app responds on port 5173.

**Acceptance Scenarios**:

1. **Given** the updated `frontend/Dockerfile`, **When** the image is built, **Then** the build completes without errors and pnpm is used for dependency installation.
2. **Given** a running container built from the updated image, **When** a request is made to port 5173, **Then** the frontend application responds correctly.
3. **Given** the full `podman-compose up --build` command executed from the project root, **When** all services start, **Then** the frontend is accessible at `http://localhost:5173` and reports healthy.

---

### User Story 4 - AGENTS.md documents pnpm as the package manager (Priority: P4)

The `frontend/AGENTS.md` file is updated to reflect pnpm as the package manager, so that future automated agents and developers have the correct context for working on the frontend.

**Why this priority**: Documentation correctness ensures agents and contributors do not default to npm commands, preventing future conflicts and misdirected agentic actions.

**Independent Test**: Read `frontend/AGENTS.md` and confirm pnpm is listed as the package manager with relevant usage instructions and no npm references remain.

**Acceptance Scenarios**:

1. **Given** the updated `frontend/AGENTS.md`, **When** an agent or developer reads it, **Then** pnpm is clearly identified as the package manager for the frontend project.
2. **Given** `frontend/AGENTS.md`, **When** it is reviewed, **Then** no npm-specific installation or script commands are present.

---

### Edge Cases

- What happens when a developer has npm but not pnpm installed? The `frontend/AGENTS.md` must document how to install pnpm before working on the project.
- How does the system handle a stale `node_modules` previously generated by npm? Migration steps must note that `node_modules` should be removed before running `pnpm install` to avoid conflicts.
- What if a future CI/CD pipeline assumes npm? The `frontend/AGENTS.md` must make pnpm usage explicit so pipeline configuration is authored correctly from the outset.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The `frontend/` directory MUST contain a `pnpm-lock.yaml` file as the sole lockfile for dependency resolution.
- **FR-002**: The `frontend/package-lock.json` file MUST be removed from the repository.
- **FR-003**: The `frontend/Dockerfile` MUST install pnpm and use it for all dependency installation and application startup steps.
- **FR-004**: All npm-specific commands in `frontend/AGENTS.md` MUST be replaced with their pnpm equivalents.
- **FR-005**: The `frontend/AGENTS.md` MUST document pnpm as the official package manager, including how to install pnpm and how to run common development tasks.
- **FR-006**: The full `podman-compose up --build` workflow from the project root MUST succeed end-to-end after the migration.
- **FR-007**: Existing `package.json` scripts (`dev`, `build`, `preview`) MUST remain functional when invoked via pnpm.
- **FR-008**: Zero references to `npm install` or `npm run` MUST remain in the `frontend/` directory after migration.

### Key Entities

- **pnpm-lock.yaml**: The new lockfile that records the exact resolved dependency tree; replaces `package-lock.json`.
- **frontend/Dockerfile**: Container build definition updated to install and invoke pnpm.
- **frontend/AGENTS.md**: Agent and developer context file updated to reflect pnpm as the package manager.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `pnpm install` completes with zero errors in the `frontend/` directory.
- **SC-002**: `pnpm run dev` starts the development server and the application is reachable at `http://localhost:5173` within the normal startup time.
- **SC-003**: `podman-compose up --build` completes successfully and all services are healthy.
- **SC-004**: Zero references to `npm install`, `npm run`, or `package-lock.json` remain anywhere in the `frontend/` directory after migration.
- **SC-005**: `frontend/AGENTS.md` contains a dedicated section describing pnpm as the package manager with installation and usage instructions.

## Assumptions

- The project currently uses npm as the package manager for the frontend, evidenced by the presence of `package-lock.json` and `npm install` in `frontend/Dockerfile`.
- The base Docker image (`node:22-alpine`) does not ship with pnpm by default; pnpm will be enabled via `corepack enable pnpm` or installed via a separate step in the Dockerfile.
- This migration is scoped exclusively to the `frontend/` directory; the backend and classifier services use Python and are unaffected.
- The existing `package.json` scripts do not need to change; only the invocation method changes from npm to pnpm.
- The `node_modules` directory is already git-ignored and does not require removal from version control history.
- No CI/CD pipeline configuration files exist in the repository at this time; if added in the future, they must use pnpm.
